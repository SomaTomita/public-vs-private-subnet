# 03. ルーティング（ルートテーブル・BGP・デフォルトルート）

> このドキュメントが答える問い：「VPCの中の通信はどうやって行き先を決めているのか？ `terraform/routing.tf` は何を書いているのか？」

## ルーターの役割

ルーターは **「異なるネットワーク間でパケットを転送する装置」** です。同じサブネット内の通信ならスイッチだけで済みますが、違うサブネット・違うVPC・インターネットへ出るには必ずルーターを経由します。

## ルートテーブル＝GPS

ルーターは **ルートテーブル（forwarding table）** という地図を持ち、受信したパケットの宛先IPを照らし合わせて「次にどこへ送るか」を決めます。

判断ロジックは単純です：

```
パケット受信
   ↓
宛先IPに一致するエントリはあるか？
   ├── Yes → そのエントリ通りに転送
   └── No  → デフォルトルート(0.0.0.0/0)はあるか？
              ├── Yes → デフォルトルートに従う
              └── No  → 黙って破棄（ICMPエラーは返さない）
```

> ルーターは **絶対に「推測」しない**。テーブルに書かれていなければ捨てるだけ。「Webサイトが表示されない」の多くはルートテーブルの設定漏れが原因です。

## ルートテーブルの最小構成

| 宛先 (Destination) | ターゲット (Next hop / Target) | 意味 |
|---|---|---|
| `10.0.0.0/16` | `local` | このVPC/LAN内は自分でルーティングできる |
| `0.0.0.0/0`   | `igw-xxxx` | それ以外はすべてインターネットゲートウェイへ（デフォルトルート） |

**最も具体的な（プレフィックスが長い）エントリが優先**されます (longest prefix match)。`10.0.0.0/16` は `0.0.0.0/0` より具体的なので先に当たります。

## デフォルトルート（`0.0.0.0/0`）

「テーブルに無い宛先のための最後の砦」。インターネット全体を 1 行で表現する魔法の記法です。

これがないと：

- 家庭用ルーターは「世界中のWebサイトのIP範囲」を全部書く必要があり破綻する
- VPCのEC2はインターネット上の `8.8.8.8` に通信できない

逆に **デフォルトルートを `IGW` ではなく `NAT Gateway` に向ければ**、そのサブネットは「外向きしか出られないプライベートサブネット」になります。これが Config B の核心です。

## 静的ルート vs 動的ルート

### 静的ルート（Static routing）

人間が手動で書く。小規模ネットワークやクラウドの一般的な設計（VPCは基本これ）。
**本リポジトリのルートテーブルもすべて静的**です。

### 動的ルートプロトコル

ルーター同士が会話してルート情報を自動交換する。

| プロトコル | 主な用途 |
|---|---|
| **BGP** (Border Gateway Protocol) | インターネットの基幹。ISP同士が経路を交換する |
| OSPF | 企業内の大規模LAN |
| RIP | 古典的、教育用 |

AWS では **AWS Direct Connect / Site-to-Site VPN** で BGP を使うことがあります。VPC内のルーティングだけなら BGP は不要。

## このリポジトリでの実践：`terraform/routing.tf`

ルートテーブルは以下のように構成されています（[architecture.md](../../architecture.md) より）。

### Public ルートテーブル

| 宛先 | ターゲット |
|---|---|
| `10.0.0.0/16` | `local` |
| `0.0.0.0/0`   | `igw-xxxx` ← Internet Gateway |

→ Public サブネット（`10.0.1.0/24`, `10.0.3.0/24`）に関連付けられる。**インターネットと双方向通信できる。**

### App Private ルートテーブル

`config_mode = "private"` の場合：

| 宛先 | ターゲット |
|---|---|
| `10.0.0.0/16` | `local` |
| `0.0.0.0/0`   | `nat-xxxx` ← NAT Gateway |

→ **外向きのみ通る**。インターネットからの直接アクセスは不可。

`config_mode = "public"` の場合は EC2 が Public 側に置かれるため、App Private ルートテーブルは存在するが EC2 からは使われない。

### DB Private ルートテーブル

| 宛先 | ターゲット |
|---|---|
| `10.0.0.0/16` | `local` |

→ **デフォルトルートなし**。RDS は VPC 内通信のみ。完全に外部から隔離される最も厳しい設定。

## サブネットとルートテーブルの関係

- **1 つのサブネット → 1 つのルートテーブル**（同時には）
- **1 つのルートテーブル → 複数のサブネット**（共有可）
- 関連付けを切り替えれば、サブネットの性質（パブリック⇔プライベート）が即座に変わる

これが、`config_mode` を切り替えるだけで攻撃面が変わる仕組みの本質です。

## 確認用：実際のルートテーブルの読み方

`terraform output` の後、AWS マネジメントコンソールで「VPC → Route tables」を開き、Routes タブを見れば上記の表が実物として確認できます。

```bash
# どのサブネットにどのルートテーブルが付いているか
aws ec2 describe-route-tables \
  --filters "Name=vpc-id,Values=$(terraform output -raw vpc_id)" \
  --query 'RouteTables[].{Name:Tags[?Key==`Name`]|[0].Value,Associations:Associations[].SubnetId,Routes:Routes[].{Dst:DestinationCidrBlock,Tgt:GatewayId||NatGatewayId}}'
```

## ハマりどころ

- **IGW を作っただけではインターネットに出られない**。サブネットのルートテーブルに `0.0.0.0/0 → igw-xxxx` を書いて初めてパブリックサブネットになる
- **`local` ルートは削除も編集もできない**。AWSが管理する
- **Private サブネットから S3 を呼ぶときは NAT Gateway を経由するのではなく、VPC Endpoint（Gateway型）を使うとコストゼロで済む**。本リポジトリでは `S3 Gateway Endpoint` を採用済み（`architecture.md` 参照）
