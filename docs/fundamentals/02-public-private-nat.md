# 02. パブリックIP・プライベートIP・NAT・IANA階層

> このドキュメントが答える問い：「`10.0.0.0/16` ってインターネットで使えないIPでしょ？ なぜVPCで成立するの？ そしてプライベートな端末がどうやって外に出ていくの？」

## パブリックIPとプライベートIPの違い

| 項目 | パブリックIP | プライベートIP |
|---|---|---|
| インターネットでの可視性 | 見える | 見えない（直接ルーティング不可） |
| 用途 | Webサーバー、家庭用ルーターのWAN側、企業の出口 | 家庭・オフィス内、データセンター内、VPC内 |
| グローバル一意 | はい（世界に1つだけ） | いいえ（誰でも使い回せる） |
| 取得方法 | ISP・クラウド・IANA階層から払い出し | RFC1918 から好きに使う |

> インターネット上で「届く」ためには、**送信元・宛先の両方がパブリックIPで識別できる**必要があります。郵便で言えば、住所が国の郵便システムに登録されていなければ手紙は届かない、と同じことです。

## RFC1918：誰でも使えるプライベート範囲

[RFC 1918](https://datatracker.ietf.org/doc/html/rfc1918) で定義された 3 つの範囲は、**誰の許可もなく自由に**社内・家庭内ネットワークで使えます（その代わりインターネットでは流せません）。

| 範囲 | クラス | 規模 | 主な用途 |
|---|---|---|---|
| `10.0.0.0/8`        | A | 約 1,677 万 IP | 大企業・データセンター・**AWS VPC** |
| `172.16.0.0/12`     | B | 約 100 万 IP   | 中規模ネットワーク |
| `192.168.0.0/16`    | C | 65,536 IP      | 家庭用ルーター |

**本リポジトリの `10.0.0.0/16` は `10.0.0.0/8` の中**。だからインターネットで衝突せず、VPC 内で自由に使えます。

## NAT：プライベートIPがどうやってインターネットに出るか

家庭の Wi-Fi ルーターを思い出してください。

```
[家のPC]              [Wi-Fiルーター]              [インターネット]
192.168.1.20  ───→   192.168.1.1 / 100.255.0.3  ───→   cnn.com
                     (LAN側プライベート / WAN側パブリック)
```

ルーターが行う **NAT (Network Address Translation)** の仕事：

1. **送信時**：パケットの送信元IPを、PCの `192.168.1.20` から自分のパブリックIP `100.255.0.3` に書き換えてインターネットに流す
2. **受信時**：戻ってきた応答パケットの宛先IPを、`100.255.0.3` から `192.168.1.20` に戻して PC に届ける

これにより、**1 つのパブリックIPの裏に何十台ものプライベート機器**を隠せます。IPv4 アドレスの枯渇を引き延ばした立役者です。

> AWS の場合、EC2 インスタンスに「パブリックIP」が表示されていても、**実際にはインスタンス自身の NIC にはついていません**。Internet Gateway (IGW) または NAT Gateway 上で NAT 変換が行われています。これが `ip addr` でパブリックIPが見えない理由です。

## IANA階層：誰がパブリックIPを払い出すのか

世界中で衝突せず一意になるよう、パブリックIPは階層的に管理されています。

```
IANA (Internet Assigned Numbers Authority)
  │ ※グローバルな最上位機関
  ├── ARIN     (北米)
  ├── RIPE NCC (欧州・中東)
  ├── APNIC    (アジア・太平洋)         ← 日本はここ → JPNIC へ
  ├── LACNIC   (中南米)
  └── AFRINIC  (アフリカ)
        ↓
      各国の国別レジストリ (NIR)
        ↓
      ISP (インターネットサービスプロバイダー)
        ↓
      エンドユーザー（あなた、企業）
```

電話番号の「国番号 → 市外局番 → 個人番号」と似た仕組みです。

## このリポジトリでの実践

### VPC内のIPはすべてプライベート

`terraform/main.tf` の VPC 定義：

```
cidr_block = "10.0.0.0/16"   ← RFC1918 のクラスA
```

EC2 インスタンスが受け取る `10.0.10.x` のようなIPは**すべてプライベート**。インターネットからは直接見えません。

### Config A（Public モード）でなぜEC2に「パブリックIP」が付くのか

Public サブネットに置かれた EC2 には、AWS が自動でパブリックIPを払い出します。ただしこのIPは **IGW 上での NAT 変換用**であり、OS から見ると NIC にはプライベートIPしか付いていません。

```bash
# EC2 にログインして確認
ip addr   # → 10.0.1.x（プライベートしか見えない）
curl ifconfig.me   # → グローバルから見えるパブリックIP
```

### Config B（Private モード）の NAT Gateway

`terraform/nat.tf` で作成される NAT Gateway は、**プライベートサブネットの EC2 がインターネットに「出ていく」ための装置**です。家庭の Wi-Fi ルーターと同じ役割を AWS のマネージドサービスとして提供しています。

- **outbound（外向き）のみ許可**：EC2 → インターネットは通る
- **inbound（内向き）はブロック**：インターネット → EC2 は通らない
- そのため Config B では、**IGW を経由した攻撃者の直接スキャンが届かなくなる**

→ これが [architecture.md の Attack Surface Matrix](../../architecture.md) で `Network Boundary: BLOCKED` になる仕組みです。

## ハマりどころ

- **「パブリックIPが付いている = インターネットから見える」ではない**。サブネットがパブリック設定（IGW へのルートあり）でなければ意味がない（→ [07. パブリックサブネット vs プライベートサブネット](./07-public-private-subnet.md)）
- **NAT Gateway は egress（外向き）フィルタリングをしない**。マルウェアが C2 サーバーへ通信する経路としても使えてしまう（→ `architecture.md` の "Required complementary controls"）
