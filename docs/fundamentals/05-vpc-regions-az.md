# 05. AWS VPC・リージョン・AZ（仮想データセンター）

> このドキュメントが答える問い：「VPCはリージョンに閉じる、サブネットはAZに閉じる——なぜそんな制約がある？ なぜ本ラボはAZ-aとAZ-cの両方にサブネットを切るのか？」

## AWS グローバルインフラの3階層

```
リージョン (Region)
  ├── アベイラビリティゾーン (AZ) #1
  │     ├── データセンター
  │     ├── データセンター
  │     └── データセンター
  ├── アベイラビリティゾーン (AZ) #2
  │     └── データセンター ...
  └── アベイラビリティゾーン (AZ) #3
        └── データセンター ...
```

| 階層 | 単位 | 物理的な意味 |
|---|---|---|
| **リージョン** | 国・地域 | 例：東京 (`ap-northeast-1`)、北バージニア (`us-east-1`) |
| **AZ** | 1 つ以上のデータセンター群 | リージョン内で物理的に離れている（数 km〜数十 km）|
| データセンター | サーバーラック | 1 AZ に複数あり、超高速回線で結合 |

2026 年時点で AWS は 38 以上のリージョン、120 以上の AZ を持っています（増加中）。

## AZ が「複数あること」の意味

> 1 つの AZ が落ちても、別の AZ が生きていれば、サービスは続く。

これが AWS のクラウドネイティブ設計の根本思想です。AZ 同士は**地理的に離れた別の建物**で、超高速回線で繋がっているため、リージョン内のレプリケーションは「同一AZ並みに速く」「障害には独立」となります。

公式の設計指針：**プロダクション ワークロードは複数AZに分散させること**。これが「Multi-AZ 構成」「高可用性 (HA)」の正体です。

## VPC：自分専用の仮想データセンター

VPC (Virtual Private Cloud) は、AWS リージョン内に作る**論理的に隔離されたネットワーク空間**です。

| 性質 | 内容 |
|---|---|
| 範囲 | **1 リージョンに閉じる**（リージョンをまたぐVPCは作れない） |
| AZ | 1 つの VPC は**そのリージョンの全 AZ にまたがって存在**できる |
| 隔離 | デフォルトで他のVPC・他アカウントから完全隔離 |
| 構成要素 | CIDRブロック、サブネット、ルートテーブル、IGW、NACL、Security Group など |
| デフォルトVPC | アカウント作成時に**全リージョンに自動で1つ**作成される |

### サブネットは AZ に閉じる

これが VPC 設計で最初に覚えるべき制約：

> **1 つのサブネットは、1 つの AZ に閉じる。** AZ-a に作ったサブネットの IP を AZ-c のホストに割り当てることはできない。

複数 AZ に同じ役割（例：App 層）を置きたければ、**AZごとに別のサブネットを切る**必要があります。

## VPC の最小構成要素

| 要素 | 役割 | 本ラボでの場所 |
|---|---|---|
| **CIDR ブロック** | VPCのIP範囲（例：`10.0.0.0/16`） | `terraform/main.tf` |
| **サブネット** | CIDRを切り分けたAZ単位の小範囲 | `terraform/locals.tf`, `main.tf` |
| **暗黙のルーター** | 全サブネットを繋ぐ。AWSが自動管理 | （コードに登場しない） |
| **ルートテーブル** | 「この宛先IPはここへ」をルーターに教える | `terraform/routing.tf` |
| **Internet Gateway (IGW)** | VPCをインターネットに繋ぐ唯一の出入口 | `terraform/main.tf` |
| **Security Group / NACL** | 仮想ファイアウォール（次層で扱う） | `terraform/security_groups.tf` |

CIDR は `/16`（最大）から `/28`（最小）まで。AWS のデフォルトは `/16` で、サブネットも最大 `/28` まで分割可能です。

## このリポジトリでの実践

`architecture.md` の **CIDR Design** をこの観点で読み直すと、AZの意味が見えます。

```
VPC (10.0.0.0/16) ─── ap-northeast-1
  │
  ├── AZ-a (ap-northeast-1a)
  │     ├── 10.0.1.0/24    Public
  │     ├── 10.0.10.0/24   App Private
  │     └── 10.0.20.0/24   DB Private
  │
  └── AZ-c (ap-northeast-1c)
        ├── 10.0.3.0/24    Public
        ├── 10.0.11.0/24   App Private
        └── 10.0.21.0/24   DB Private
```

### なぜ AZ-a と AZ-c の両方？

| 理由 | 解説 |
|---|---|
| **ALB の要件** | ALB は最低 2 AZ を要求する。Config B では Public サブネットが両 AZ に必要 |
| **RDS Subnet Group の要件** | RDS は Multi-AZ 構成のために必ず 2 AZ 以上のサブネットが必要（実機は 1 AZ にしかなくとも） |
| **将来の冗長化** | EC2 を Auto Scaling で複数 AZ に分散させる前提を維持 |

ラボでは EC2 自体は 1 台しか起動しませんが、**サブネット設計は本番運用の型を踏襲**しています。

### リージョンを変えると何が起きる？

`terraform.tfvars` で `aws_region = "us-east-1"` に変えるだけで、北バージニアにまるごとデプロイし直せます。これが「VPCはリージョンに閉じる」設計の利便性。

## ハマりどころ

- **「VPCはリージョンに閉じる」≠「リージョン跨ぎの通信ができない」**。VPC Peering や Transit Gateway を使えば別VPC（別リージョン含む）と接続可能
- **AZ名 (`a`, `b`, `c`) は AWS アカウントごとにマッピングが違う**。あなたの「ap-northeast-1a」と他人の「ap-northeast-1a」は別の物理AZの可能性あり
- **新規アカウント作成時のデフォルトVPCは便利だが、本番では消して自前でVPCを設計するのが定石**。本ラボもデフォルトVPCを使わず `terraform/main.tf` で明示的に定義している
