# 01. IPアドレスとサブネッティング（ピザの例え）

> このドキュメントが答える問い：「`10.0.0.0/16` の `/16` って結局なに？ なぜ `/24` に切るの？」

## IPアドレスは 32 ビットの数字

IPv4 アドレスは 32 ビット長で、人間に分かりやすいよう 8 ビット（オクテット）ごとに `.` で区切って表記します。

```
10.0.1.0
└┬┘ └┬┘ └┬┘ └┬┘
 8b   8b   8b   8b   合計32ビット
```

この 32 ビットは **ネットワーク部** と **ホスト部** に分かれます。

- **ネットワーク部**：「どのネットワークに属するか」を表す
- **ホスト部**：「そのネットワーク内のどの端末か」を表す

## サブネットマスクと `/n` 表記

「先頭から何ビットがネットワーク部か」を示すのが **サブネットマスク** です。CIDR 表記では `/n` で書きます。

| 表記 | ネットワーク部 | ホスト部 | ホスト総数 |
|---|---|---|---|
| `/16` | 16 ビット | 16 ビット | 65,536 |
| `/20` | 20 ビット | 12 ビット | 4,096 |
| `/24` | 24 ビット | 8 ビット | 256 |
| `/26` | 26 ビット | 6 ビット | 64 |
| `/30` | 30 ビット | 2 ビット | 4 |

「ホスト総数 = 2^(ホスト部のビット数)」です。実際に使えるのは **総数 − 2**（ネットワークIDとブロードキャストIPは予約）。AWS ではさらに 3 つ予約されるため、`/24` の AWS サブネットで使えるのは 251 個です。

## ピザの例え（サブネッティング）

サブネッティング = **大きなピザを切り分ける**こと。

- 元のピザ：`192.168.240.0/24` （ホスト部 8 ビット、合計 256 個）
- 4 等分したい → 1 ピース 64 個

ここでルールが効いてきます：

> **4 つに分けるには、ホスト部から 2 ビットを「借りる」必要がある**。
> （借りたビットの組み合わせ `00 / 01 / 10 / 11` で 4 つのサブネットを区別する）

結果：

| サブネット | 範囲 | サブネットマスク |
|---|---|---|
| 1番目 | `192.168.240.0`   〜 `192.168.240.63`  | `/26` |
| 2番目 | `192.168.240.64`  〜 `192.168.240.127` | `/26` |
| 3番目 | `192.168.240.128` 〜 `192.168.240.191` | `/26` |
| 4番目 | `192.168.240.192` 〜 `192.168.240.255` | `/26` |

ピザを 4 等分すれば「1 切れ＝丸ごと 1 枚」にはならない。**サブネットを切ったら、元の `/24` をそのまま使うことはできない**——これが鉄則です。

## 「いくつに分けたいか」と借りるビット数

| 必要なサブネット数 | 借りるビット数 | 元が `/24` なら |
|---|---|---|
| 2 | 1 | `/25` |
| 3〜4 | 2 | `/26` |
| 5〜8 | 3 | `/27` |
| 9〜16 | 4 | `/28` |

サブネットを増やすほど 1 サブネットあたりのホスト数は減ります。設計時は **「サブネットいくつ × ホスト数いくつ」** を先に決めること。

## 重要なIPの計算

ある CIDR が与えられたとき、最低限これだけ出せれば設計に困りません。

| 求めたいもの | 求め方 |
|---|---|
| ネットワークID | ホスト部を全部 0 にした値（範囲の最小） |
| ブロードキャストIP | ホスト部を全部 1 にした値（範囲の最大） |
| 利用可能な最初のIP | ネットワークID + 1 |
| 利用可能な最後のIP | ブロードキャストIP − 1 |
| 利用可能ホスト数 | 2^(ホスト部) − 2 （AWSなら −5） |

## このリポジトリでの設計

[architecture.md](../../architecture.md) の **CIDR Design** を読み解きます。

```
VPC: 10.0.0.0/16          ← 大きなピザ（65,536 IP）
├── Public  AZ-a: 10.0.1.0/24    ← 256 IP の薄切り
├── Public  AZ-c: 10.0.3.0/24
├── App Pri AZ-a: 10.0.10.0/24
├── App Pri AZ-c: 10.0.11.0/24
├── DB  Pri AZ-a: 10.0.20.0/24
└── DB  Pri AZ-c: 10.0.21.0/24
```

### なぜ `/16` を `/24` に切るのか

- **役割を分離**：Public / App Private / DB Private を別サブネットにし、Security Group やルートテーブルで層ごとに制御する（多層防御）
- **AZ ごとに分散**：同じ役割のサブネットを AZ-a と AZ-c に持つことで、片方の AZ 障害でもサービス継続できる
- **`/24`（256 IP）で十分**：このラボの規模なら 250 IP 強あれば足りる。`/20`（4,000 IP）まで取っても無駄になる

## 自分で手を動かしてみる

紙とペンで OK。`172.16.50.0/20` について以下を計算してみてください：

1. ホスト部のビット数
2. ネットワークID と ブロードキャストIP
3. 利用可能なホスト数
4. これを 4 サブネットに分割すると、各サブネットの CIDR は？

→ 答えは ChatGPT や CIDR 計算機で検算できます。**間違えた箇所を探して直すことで、サブネッティングの感覚が定着します**。
