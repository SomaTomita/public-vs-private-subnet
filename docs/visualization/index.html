    1. OSI モデルと「層」の概念 3 用語

    ネットワーク通信は複数の層 (Layer) で重なり合っている。攻撃と防御は「どの層」で起きるかで意味が変わる。本ラボで頻出するのは L3 / L4 / L7。

    L3 (Network Layer)

    IP アドレスでホストを識別し、ルーティングする層。ICMP (ping) もここ。「L3 不可達」とは IP 的にパケットが相手に届かない状態 (例: Private Subnet には IGW 経由のルートがないため、外部 IP からのパケットが配送経路を持たない)。

    なぜ重要か: Private 配置の本質は「L3 でそもそも届かない」こと。SG/NACL より手前で防御が成立する。

    L4 (Transport Layer)

    TCP / UDP。ポート番号でアプリを識別 (22=SSH, 80=HTTP, 5432=PostgreSQL)。TCP は接続前に 3-way handshake (SYN → SYN-ACK → ACK) を行う。ポートスキャナはこの handshake の応答有無で「ポートが開いているか」を判定。

    なぜ重要か: ポートスキャンの結果 (open / closed / filtered) は L4 で決まる。SG ドロップは「filtered」を生成する。

    L7 (Application Layer)

    HTTP, HTTPS, DNS, SSH などのプロトコル本体。ヘッダや URL の中身が見える。ALB の routing や WAF のシグネチャマッチはここで動作。

    なぜ重要か: SQLi / XSS / SSRF など Web 攻撃は L7 でしか検出できない。WAF が L7 で機能する所以。

    2. AWS VPC ネットワーキングの基礎 7 用語

    AWS の VPC は IP 空間を独自に持つ仮想ネットワーク。「Public / Private」の区別はリソース自体ではなく、Route Table の中身で決まる。

    VPC

    AWS 上で作る独立した仮想ネットワーク。10.0.0.0/16 のような CIDR を持つ。

    なぜ重要か: Public/Private の議論はすべて 1 つの VPC 内のサブネット配置の話。

    Subnet

    VPC を分割した小区画。「Public」「Private」はラベルではなく、Route Table が IGW へのルートを持つかどうかで実質決まる。

    なぜ重要か: 名前やタグを「Public」と付けても Route が IGW なしなら Private 扱い。逆も然り。

    IGW (Internet Gateway)

    VPC をインターネットと接続する論理デバイス。Subnet が IGW へのデフォルトルートを持ち、かつホストに Public IP が付与されると外部から到達可能になる。

    なぜ重要か: IGW + Public IP + Route の 3 点セットが揃わなければ「公開」にならない。

    NAT Gateway

    Private Subnet 内のホストが outbound のみインターネットに出るためのデバイス。送信元 IP を NAT GW の EIP に書き換える。inbound は通さない。destination の中身を見ないので egress フィルタにはならない。

    なぜ重要か: NAT GW は「出口の集約」であって「出口の検閲」ではない。C2 / exfil の検出には別の仕組みが要る。

    Route Table

    Subnet ごとに紐づく経路表。「0.0.0.0/0 → IGW」を持つ Subnet は Public、「0.0.0.0/0 → NAT GW」 もしくはルートなしは Private。

    なぜ重要か: Public / Private の正体はここに書かれている。Terraform レビューの最重要ポイント。

    AZ (Availability Zone)

    リージョン内の独立データセンター群。ALB は最低 2 AZ をまたぐ Subnet を要求する。

    なぜ重要か: ALB を組むには「Private Subnet を 2 AZ 用意」が前提。設計初期に効いてくる。

    Public IP

    インスタンスに付与される外向き IP。これがないとインターネットからの到達対象になり得ない。

    なぜ重要か: Public IP の有無は「攻撃面が存在するか」を決める一次変数。

    3. ファイアウォール — Security Group と NACL 5 用語

    AWS の主要な FW は Security Group (SG) と Network ACL (NACL) の 2 種類。本ラボでは SG が中心。 ALB の介在による「SG ミス吸収」効果はここで理解する。

    Security Group (SG)

    ENI (≒インスタンス NIC) 単位で動作するステートフルファイアウォール。「ステートフル」とは inbound で許可した接続の戻りトラフィック (outbound) が自動的に通る性質。

    なぜ重要か: ほとんどの AWS FW 設定は SG。ここの誤りは即「全世界に開放」を意味する。

    Stateful FW

    接続単位を追跡する FW。一度許可した接続の戻りパケットを自動許可する。SG はこれ。

    なぜ重要か: outbound の許可ルール作成を省略でき、設定ミスが減る。

    Stateless FW

    各パケットを独立に評価する FW。戻りも明示的に許可する必要がある。NACL はこれ。

    なぜ重要か: NACL の rule 設計ミスは「片方向だけ通る」という奇妙な障害を生む。

    Referenced SG Rule

    Source/Destination を「他の SG ID」として指定する書き方 (例: ALB SG を許可)。 IP/CIDR ではなく論理グループで繋がるため、IP 変更に追従し、SG ミス (0.0.0.0/0 を誤許可) のリスクが下がる。

    なぜ重要か: Config B の安全性は「EC2 の SG が ALB SG のみを許可」で担保される。

    NACL (Network ACL)

    Subnet 単位で動作するステートレスフィルタ。SG の上位の "broad deny" 用途。

    なぜ重要か: 既知 IoC IP の Subnet 単位ブロックなど、SG では書きにくいルールを担う。

    4. 偵察 (Reconnaissance) と公開資産の発見 5 用語

    攻撃の最初の段階は「対象を見つける」こと。 Public IP は インターネット全体の常時スキャンに晒されている ことを理解する。

    Port Scanner

    対象ホストの開放ポートを列挙するツール (nmap, masscan, ZMap)。 TCP SYN を送り、 SYN-ACK が返れば「open」、 RST なら「closed」、 無応答なら「filtered (FW でドロップ)」と判定。

    なぜ重要か: 攻撃の起点。スキャンで見つけた open ポートが脆弱性試行のターゲットになる。

    masscan / ZMap

    インターネット全体 (IPv4 約 40 億 IP) を数十分でスキャンできる超高速スキャナ。攻撃者は常時これで世界をスキャンしている。

    なぜ重要か: 「うちは小さいから狙われない」は通じない。 Public IP がある = 数時間以内に発見される前提で設計する。

    Shodan

    インターネットを継続的にスキャンして公開資産を検索可能にしたサービス。「banner grabbing」も行うので OS や CMS のバージョンまで特定される。

    なぜ重要か: 攻撃者は port:5432 country:JP 等で標的を秒で絞り込める。

    Censys

    Shodan と同種の検索サービス。 TLS 証明書の解析にも強い。

    なぜ重要か: 証明書の SAN から内部用ホスト名がリークし、想定外の偵察起点になる。

    OSINT (Open Source Intelligence)

    公開情報からの偵察。 DNS, GitHub, SNS, 証明書透明性ログ (CT log) など。

    なぜ重要か: スキャンより手前に漏れる情報 (リポジトリの IaC, 履歴の secret 等) が多い。

    5. 攻撃の類型 — 名前を付けて整理する 12 用語

    本ラボで言及される代表的攻撃類型を整理。「層」と「目的」を区別。

    Brute Force

    ID/PW を総当たりで試す攻撃。 SSH(22) や RDP(3389) に対して hydra, medusa などで実行される。

    なぜ重要か: 22 を世界に開けると「いずれ来る」ではなく「常時試行されている」状態になる。

    Credential Stuffing

    既にどこかでリークした ID/PW の組を別サービスで試す。パスワード使い回しを狙う。

    なぜ重要か: ログイン経路を持つ全サービスの脅威。MFA / レート制限が必須対策。

    DDoS

    多数の攻撃元からトラフィックを集中させてサービス停止を狙う。 L3/L4 (SYN flood, UDP flood) と L7 (HTTP flood) がある。

    なぜ重要か: ALB + AWS Shield で L3/L4 はかなり吸収できるが、L7 は WAF と rate-based rule 設計が要る。

    Zero-day

    公開されていない未パッチ脆弱性。「ゼロデイ漁り (zero-day hunting)」とは攻撃者がスキャンで見つけた公開資産を未公表脆弱性で侵害すること。

    なぜ重要か: パッチ運用では防げない領域。攻撃面そのものを減らす Private 配置の意義はここにある。

    CVE

    公開済み脆弱性に振られる識別番号 (例: CVE-2021-44228 = Log4Shell)。 WAF はこれら既知パターンをシグネチャ化して防御する。

    なぜ重要か: 既知 CVE は「公開直後 24h で攻撃自動化される」前提で備える。

    SQL Injection

    Web フォーム等から不正な SQL 文字列を注入し、 DB の意図しない操作を行う攻撃 (' OR '1'='1 等)。 OWASP Top 10 の常連。

    なぜ重要か: L7 攻撃の代表。 Private 配置は無力で、パラメータバインドと WAF が要る。

    XSS

    攻撃者の JavaScript を被害者ブラウザで実行させる攻撃。 Reflected / Stored / DOM-based の 3 種。

    なぜ重要か: ネットワーク層では一切防げない。出力エスケープと CSP が一次防御。

    SSRF

    サーバ側プロセスから任意 URL に HTTP リクエストを発行させる攻撃。本ラボの中核 (/fetch?url=...)。 IMDS の link-local IP に到達する経路として悪用される。

    なぜ重要か: Private 配置でも IMDS は動くので、SSRF は Public/Private 双方に等しく刺さる。

    Lateral Movement

    侵害したホストから同一ネットワーク内の別ホストへ侵入を広げる行為。

    なぜ重要か: 1 台破られた後の被害規模を決める。SG の "tier 別分割" が効く局面。

    C2 (Command & Control)

    侵害後に攻撃者が遠隔操作するための通信チャネル。 HTTPS / DNS / ICMP に偽装される。

    なぜ重要か: NAT GW では検出できない。VPC Endpoint の絞り込み・egress proxy 等の設計が問われる。

    Data Exfiltration

    機密データの外部持ち出し。 S3, Pastebin, DNS, 自前 HTTPS サーバなど。

    なぜ重要か: 最終的なビジネス被害はここで発生する。 outbound 先の whitelist が最後の砦。

    OWASP Top 10

    Web アプリの代表的 10 大リスクを定期更新するコミュニティリスト。 SQLi / XSS / SSRF などはここで定義。

    なぜ重要か: WAF Managed Rules や SAST のシグネチャが OWASP に紐づく。共通言語。

    6. ALB と WAF — L7 でできる防御 8 用語

    ALB は L7 ロードバランサで、「HTTP の中身を見て」転送先を決める。 WAF は ALB 等にアタッチして HTTP リクエストの内容を検査・拒否する仕組み。

    Load Balancer (LB)

    リクエストを複数バックエンドに振り分ける装置。 L4 LB (NLB) は IP/Port で、 L7 LB (ALB) は HTTP の中身で振り分ける。

    なぜ重要か: どの層で振り分けるかで「介在で見える情報」が変わる。WAF が機能できる前提。

    L7 Load Balancer (ALB)

    HTTP/HTTPS で動作。 SSH や DB プロトコルは流せない (= フォワードしない)。これが「ALB の介在で 22, 5432 が物理的に届かない」理由。

    なぜ重要か: ALB は「L7 以外のプロトコルを構造的に弾く」自然な防御層になる。

    Host-based Routing

    Host: api.example.com ヘッダの値で転送先を変える ALB ルール。誤設定で内部限定アプリを露出するリスクもある。

    なぜ重要か: マルチテナント設計で必須。default action の設定漏れが内部アプリ露出を生む。

    Path-based Routing

    /admin/* 等の URL パスで転送先を変える ALB ルール。

    なぜ重要か: /admin 系のパスは別 TG + 追加 IP 制限が定石。

    WAF

    HTTP リクエストの中身 (URL, ヘッダ, ボディ) をシグネチャ/ルールで検査する FW。 SQLi / XSS / 既知 CVE パターンをエッジで拒否する。 AWS WAF は ALB / CloudFront / API Gateway にアタッチ可能。

    なぜ重要か: ネットワーク層では防げない L7 攻撃の唯一の入口チェックポイント。

    AWS Managed Rules

    WAF 用の AWS 提供ルールセット (Known Bad Inputs, SQL Database, Linux OS など)。

    なぜ重要か: 自前ルール作成より早く、CVE 公開直後の自動防御に追従しやすい。

    HTTP Request Smuggling

    ALB と backend で同じリクエストを違うバウンダリで解釈させる攻撃。 CL.TE / TE.CL の不一致を悪用。

    なぜ重要か: ALB を信頼しすぎると、smuggling で WAF / ALB を素通りさせられる。

    TLS Termination

    ALB が HTTPS を復号して backend に HTTP で渡す挙動。これにより WAF は平文を検査できる。

    なぜ重要か: 「ALB の内側 = 信頼ゾーン」と誤認しないこと。 ALB-to-EC2 の TLS 再暗号化も検討対象。

    7. AWS 特有の特殊 IP / サービス 7 用語

    本ラボの中核攻撃 (SSRF → IMDS) を理解するためには、 link-local IP と IMDS、そして SSH の代替である SSM の理解が必要。

    Link-local IP (169.254.0.0/16)

    RFC 3927 で定義された「同一リンク内のみ有効」な IP レンジ。ルータを越えない。 AWS は 169.254.169.254 を IMDS に予約。

    なぜ重要か: link-local は VPC Route Table の対象外。サブネット種別と無関係に EC2 内部から到達する。

    IMDS

    EC2 が自身のメタデータ (instance-id, AMI, IAM role の credential など) を取得するサービス。 169.254.169.254 で listen。 VPC routing 管轄外であり、 Subnet の Public/Private は無関係。

    なぜ重要か: SSRF + IMDSv1 で IAM credential が抜ける。 Private 化は防御にならない。

    IMDSv1

    単純な GET だけで credential が取れる方式。 SSRF と相性が悪い (脆弱)。

    なぜ重要か: 既存 EC2 が IMDSv1 のまま放置されると SSRF 1 回で IAM 権限が奪われる。

    IMDSv2

    PUT で セッショントークンを取り、 GET ヘッダに付ける必要がある方式。 GET-only SSRF を遮断できる。 http_tokens = "required" で強制。

    なぜ重要か: 必須化が SSRF 対策の決定打。新規構築では default required にする。

    SSM Session Manager

    SSH を使わず AWS IAM 認証で EC2 に接続する仕組み。 22 ポートを開ける必要がない。 ssm-agent が outbound で SSM endpoint と通信し、双方向セッションを確立。

    なぜ重要か: 22 を世界に開ける必要がなくなり、Private 配置でも管理可能になる。

    VPC Endpoint

    AWS API トラフィックを VPC 内に閉じる仕組み。 Gateway Endpoint (S3 / DynamoDB) と Interface Endpoint (それ以外) がある。

    なぜ重要か: outbound を VPC 外に出さずに AWS API を呼べる = exfil 経路を絞れる。

    aws:SourceVpc / aws:SourceVpce

    IAM ポリシーの条件キー。「この VPC からの API 呼び出しのみ許可」と指定でき、盗難認証情報を VPC 外で無効化できる。

    なぜ重要か: credential が漏れても「うちの VPC 外では使えない」という最後の鍵になる。

    Section 5 / Hybrid

    もし Config A に ALB を足したら? — ALB 単独 vs Private 配置の本質

    "ALB を入れれば安全" は誤解。 ALB が提供する防御層 と Private 配置が提供する防御層 は別物で、互いに代替できません。Public Subnet に置いたままの EC2 に ALB を前置きしたハイブリッド構成では、何が変わって何が変わらないかを 並べて確認します。

    Config A (Public 直接) / Hybrid (Public + ALB) / Config B (Private + ALB) の3列比較
    観点 Config A
    Public 直接     		Hybrid
    Public + ALB     		Config B
    Private + ALB

    ALB は "L7 強制 + WAF 挟込" の防御層。Private 配置は "L3 不可達 + Public IP 消去" の防御層。これらは互いに代替できない。

    ALB だけを足しても、EC2 が Public Subnet にあって Public IP を持つ限り、攻撃者にとっての「対象 IP の発見」「ポートスキャンへの応答」は変わらない。 Private Subnet 配置の本質は、ネットワーク経路自体を切ることである。

    なぜ Hybrid 構成が実務で観測されるか

    • レガシー移行中の中間状態 (EC2 の Public IP を一括で外せない事情)
    • "ALB を入れれば安全" という誤解に基づく構成
    • SaaS / 外部統合の都合で IP 直接アクセスを許可しているケース

    推奨: ALB を入れたら同時に行うべきこと

    1. EC2 を App Private Subnet に移動する
    2. EC2 の Public IP を外す (associate_public_ip_address = false)
    3. NAT Gateway を導入し outbound のみ許可する
    4. App SG inbound を ALB SG only に絞る
    5. SSH (22) を閉じ、SSM Session Manager に切替える

    Section 11 / Conclusion

    Key Takeaways

    Private subnet は AWS ワークロードを保護する上で 必要だが十分ではない。

    Private subnet が防げるもの

    • 外部からの直接 port scan / service enumeration
    • SSH brute-force from the internet
    • IP ベースの直接攻撃

    Private subnet が防げないもの

    • Application 層 (SSRF → IMDS → IAM)
    • IAM credential abuse (AWS API 層)
    • Outbound exfiltration (NAT は egress を絞らない)
    • VPC 内部の lateral movement

    Required complementary controls

    1. IMDSv2 enforcement (http_tokens="required")
    2. Least-privilege IAM + SCP (aws:SourceVpc)
    3. AWS WAF on ALB
    4. Secrets Manager
    5. Restrictive VPC Endpoint policies
    6. Egress filtering (Network Firewall / DNS Firewall)
    7. GuardDuty + CloudTrail
    8. ALB access logs

    Network isolation is necessary but not sufficient.

    Private subnet は最初の防御線として強力だが、application / IAM / outbound の各層には別の対策が必要。 多層防御 (defense-in-depth) を前提に設計しよう。
